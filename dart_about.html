Background

왜 필요했나

EV(전기차) 서비스 운영에서 BMS(Battery Management System) 텔레메트리 데이터는 차량 상태 모니터링의 핵심입니다. 그런데 AWS S3에 매일 수십~수백 대 분량의 Parquet 파일이 쌓이면서, 데이터 품질을 사람이 직접 눈으로 확인하는 방식은 한계에 부딪혔습니다.

🔴 기존 방식의 문제

수작업으로 CSV 열어 값 확인 → 오류 발견 시 이미 수일치 데이터가 쌓인 상태.
Fleet마다 허용 범위가 달라 규칙이 문서에만 존재 → 담당자마다 기준 해석이 달랐음.

🟢 필요했던 것

S3에서 자동 취득 → 35개 항목 일괄 검증 → 결과 리포트 생성까지
클릭 한 번으로 처리되는 도구. 엔지니어가 아닌 팀원도 결과를 바로 읽을 수 있어야 함.

"검증 규칙이 70개가 넘는데 매번 수작업으로 확인한다고? 그리고 Fleet마다 예외 규칙도 다른데 이걸 사람이 다 외워서 체크한다고? — 이게 말이 되는 상황인가."

결국 규칙을 코드로 명문화하고, 누가 실행해도 동일한 결과를 내는 자동화 도구가 반드시 필요했습니다. 그리고 그 결과물이 DART입니다.

Development

어떻게 만들었나 — 바이브코딩

DART는 전통적인 스프린트·기획 문서 없이, AI(Claude)와 대화하며 즉흥적으로 코드를 완성해가는 "바이브코딩(Vibe Coding)" 방식으로 개발됐습니다.

바이브코딩은 개발자가 "지금 이 기능이 필요해"라고 AI에게 말하면, AI가 코드를 작성하고 개발자는 그것을 검토·수정하는 방식입니다. 기획 → 설계 → 구현의 순서 대신, 아이디어 → 즉시 구현 → 피드백의 짧은 루프가 반복됩니다.

"수집 윈도우 상태 머신 구현해줘. IGN이 꺼진 다음 충전이나 슬립 대기 구간 데이터만 골라서 16번, 19번, 20번 검증에 써야 해."

→ AI가 즉시 _get_collection_window_mask() 함수를 작성하고 설명을 붙여줬습니다.

개발 진행 타임라인

Phase 1–2

기반 구조 설계

S3Handler, DataProcessor, BMSDataValidator 클래스 분리. 로깅 모듈 구축. AWS S3 Parquet 다운로드 → CSV 변환 파이프라인 완성.

Phase 3–4

35개 검증 항목 구현

A 기본 상태(1–10), B 셀 전압(11–16), C 모듈 온도(17–20), D 에너지·전력(21–27), E 복합 로직(28–35) 5개 그룹·71개 조건 코드화. vehicle_master.json 3단계 차량 매칭 도입.

Phase 5

Fleet 예외 규칙

VWGKALRT: acc_dchg 음수 허용, em_speed ≤ 255, 전류 +1A 기준. RAY: acc 항목 N/A. BOLT 2017: mile_km N/A. 데이터 구간 자동 선별 로직(IGN 기반).

Phase 6–7

PyQt5 GUI + 배치 처리

데스크톱 GUI 구현. PID 리스트 기반 100대 이상 배치 자동 검증. 통합 결과 Excel 리포트 자동 생성.

Phase 8–9

HTML 리포트 · 결과 뷰어

generate_html_report() — 브라우저에서 바로 열 수 있는 개별 결과 HTML 생성 (도넛 차트, 필터, 검색, 인쇄).
report_viewer.html — CSV/XLSX 드래그앤드롭 결과 뷰어 대시보드.

바이브코딩이란 무엇인가

기존 개발 방식

기획서 → 설계 문서 → 스프린트 → 코드 리뷰 → QA → 배포.
각 단계에서 컨텍스트 전달 비용 발생. 아이디어→실제 코드까지 수 일~수 주.

바이브코딩 방식

아이디어를 말로 설명 → AI가 즉시 코드 생성 → 개발자가 검토·방향 조정.
같은 날 안에 새 기능을 프로덕션 코드 수준으로 작성하는 것이 가능.

DART에서 바이브코딩이 특히 효과적이었던 이유는, 검증 규칙의 수가 많고 복잡하지만 패턴이 반복적이기 때문입니다. "항목 22번도 21번과 동일한 구조로, 다만 컬럼명은 curr_a이고 범위는 ±1000A야" 라고 말하면 AI가 기존 코드를 참고해 즉시 확장해줬습니다.

Key Decisions

핵심 설계 결정 3가지

① 규칙을 코드로 — 문서가 아닌 함수로

기존에는 검증 기준이 엑셀 문서에 텍스트로만 존재했습니다. DART는 모든 규칙을 Python 함수로 구현해 사람이 실행하지 않아도 항상 동일한 결과를 냅니다. 새 규칙 추가는 새 메서드 추가와 같습니다.

② Fleet/차종 예외는 데이터 기반 자동 적용

VWGKALRT, HKMC, GM, Tesla는 BMS 구조가 다릅니다. vehicle_master.json(317개 모델)으로 3단계 매칭 후 Fleet별 예외 규칙을 자동 선택합니다. 담당자가 Fleet를 기억하거나 설정할 필요가 없습니다.

③ 결과는 누구나 읽을 수 있어야 한다

엔지니어가 아닌 팀원도 검증 결과를 확인할 수 있도록, Excel 외에 브라우저에서 바로 열리는 HTML 리포트와 CSV 드래그앤드롭 결과 뷰어 대시보드를 추가했습니다. 서버나 추가 설치 없이 파일 하나로 완결됩니다.

바이브코딩으로 배운 것

DART 개발에서 AI는 단순한 코드 생성기가 아니었습니다. "왜 이 규칙이 필요한가"를 함께 생각하는 파트너였습니다. 35개 검증 항목 중 수집 윈도우 상태 머신처럼 까다로운 로직도, "시동 꺼진 뒤 충전·슬립 대기 데이터만 봐야 해" 라는 한 문장으로 출발해 실제 작동하는 코드가 됐습니다.

바이브코딩은 속도가 빠른 대신, 개발자가 결과를 꼼꼼히 검토하고 방향을 잡는 역할을 더 잘 해야 합니다. AI가 코드를 쓰고 개발자가 도메인 판단을 한다 — 이 조합이 DART를 만든 방식입니다.

Presenter Notes

발표 설명 — 핵심 메시지

아래는 발표 시 각 포인트별로 전달할 핵심 내용입니다. 슬라이드 순서와 맞춰 활용하세요.

1

도구가 없었다

전기차 관련 서비스 운영팀에서 BMS(Battery Management System) 데이터를 받아보면 품질을 검증하는 도구 자체가 없었습니다.
데이터를 S3에서 내려받아도 이상 여부를 수작업으로 확인해야 했고, 기준도 문서마다 달랐습니다.

DART는 이 문제를 해결하기 위해 만들었습니다. AWS S3에서 데이터를 받아서 실행만 하면 검증 결과가 나오는 구조를 목표로 잡았습니다.

2

개발하면서 규칙이 늘었다

처음에는 기본적인 범위 검사 몇 개로 시작했습니다. 실제 데이터를 보면서 35개 항목, 71개 조건까지 늘어났습니다.
검증 규칙이 코드로 명문화되자, 팀 내에서 "이 항목도 검사해야 하지 않나?" 하는 논의가 생기고 자연스럽게 범위가 확장됐습니다.

3

차량마다 데이터가 다르다

도구를 쓰면서 알게 된 것 — 같은 BMS 컬럼이라도 Fleet마다 값의 의미와 허용 범위가 다릅니다.
VWGKALRT는 방전 전류량이 음수로 기록됩니다. RAY는 누적 전력량 컬럼이 없습니다. Bolt 2017은 주행거리 컬럼이 없습니다.

이 예외들을 코드로 처리하지 않으면 정상 데이터를 FAIL로 판정하는 오탐이 발생합니다. vehicle_master.json으로 차량을 자동 식별해 Fleet별 규칙을 자동 선택하는 구조가 이래서 필요했습니다.

4

바이브코딩(Vibe Coding) — 2025년의 새로운 개발 방식

DART는 AI와 대화하며 코드를 만들어나가는 바이브코딩 방식으로 개발됐습니다.
기획서도, 스프린트 일정도 없이 — "이 기능이 필요해"를 AI에게 말하면 AI가 코드를 작성하고 개발자는 방향을 잡는 역할을 합니다.

"AI(클로드)와 Align — 대화 → 코드 확장 → 검토 → 반복.
이것이 바이브코딩입니다. 기존 개발보다 훨씬 빠르게 아이디어가 작동하는 코드가 됩니다.
2025년 현재 AI 도구의 가장 실용적인 활용 방식 중 하나입니다."
5

실제 예시 — AI가 작성한 핵심 로직

가장 까다로운 로직 중 하나인 수집 구간 자동 선별 함수도 바이브코딩으로 만들었습니다.
"시동 OFF 후 충전·슬립 대기 구간 데이터만 골라야 해"라는 한 마디에서 시작됐습니다.

# AI가 작성한 핵심 함수 (요약) def _get_collection_window_mask(self): """IGN OFF 후 충전·슬립 대기 구간 행만 True로 마스킹""" ign = self.df['ignit_status'] relay = self.df['main_relay_status'] # IGN 0→0 유지, relay=1 (충전/슬립 대기) 구간 선택 mask = (ign == 0) & (relay == 1) return mask # 항목 16, 19, 20에 적용

이 함수 하나로 주행 중 오탐 방지와 수집 구간 정밀 검증이 동시에 해결됩니다. AI가 코드를 쓰고 개발자가 도메인 검증을 했습니다.
